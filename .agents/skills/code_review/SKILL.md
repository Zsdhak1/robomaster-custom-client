# Flutter 代码审查技能

## 本仓库接手入口

- 审查本仓库前，先读 `feature_spec.md` 顶部「新 Agent 接手摘要」，确认当前版本、历史遗留项和本次任务标注。
- 代码发现优先使用 codebase-memory MCP；若工具不可用，或查文档/配置/字符串，再用 `rg`。
- `v0.0.1 Phase 3, Task 3.5` 是历史遗留项，不应在普通审查中当作当前阻塞问题。

## 审查范围

1. **架构合规性**
   - 数据模型是否先于 UI 定义
   - 业务逻辑是否在正确的层（domain/data，不在 presentation）
   - 是否存在跨 Feature 的直接引用
   - core/ 是否被 Feature 引用，而非反向

2. **代码质量**
   - 函数长度是否超过 50 行
   - 是否存在重复代码（与已有代码功能重复）
   - 类长度是否超过 300 行
   - 文件长度是否超过 500 行

3. **命名规范**
   - 类名：UpperCamelCase
   - 变量/函数：lowerCamelCase
   - 文件：snake_case
   - 常量：lowerCamelCase
   - 禁止中文拼音命名

4. **状态管理**
   - Widget 中是否直接修改状态（应使用 ref.read(provider.notifier)）
   - 状态读取是否使用 ref.watch（监听变化）或 ref.read（一次性读取）
   - 复杂状态是否使用 StateNotifier 而非简单 StateProvider

5. **空安全**
   - 是否存在未处理的 null
   - 是否滥用 ! 操作符
   - 可选参数是否有合理的默认值

6. **常量与魔法值**
   - 魔法数字是否提取为命名常量
   - UI 字符串是否放入常量文件
   - 颜色/尺寸是否使用 Theme 或常量

7. **错误处理**
   - async 函数是否有 try-catch
   - 错误是否有用户可见的反馈
   - Stream 是否有错误处理通道

8. **性能**
   - 列表是否使用 ListView.builder
   - 大数据集是否有分页或采样
   - 是否不必要的 Widget rebuild

## 审查输出格式

```
## 代码审查报告

### 通过项
- [x] 架构合规
- [x] 函数长度合规
...

### 待修复项
- [ ] 问题描述 → 修复建议
- [ ] 问题描述 → 修复建议
...

### 严重问题（阻塞）
- [ ] 问题描述 → 必须修复

### 修复优先级
1. P0: 阻塞问题（必须立即修复）
2. P1: 重要问题（应在提交前修复）
3. P2: 建议优化（可在后续迭代处理）
```

## 工作流

1. 读取 `feature_spec.md` 的新 Agent 接手摘要与被审查文件内容
2. 检查项目中是否已有类似功能的代码（优先 MCP，必要时 `rg`）
3. 按上述 8 个维度逐项检查
4. 生成审查报告
5. 如果存在 P0 问题，自动修复并重新审查
6. 如果只有 P1/P2 问题，报告给用户并建议修复
