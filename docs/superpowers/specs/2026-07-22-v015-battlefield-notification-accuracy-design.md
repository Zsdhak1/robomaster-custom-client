# v0.1.5 战场通知内容准确化设计

## 1. 目标与边界

v0.1.5 在现有通知运行时、通知历史和设置系统上校正规则判断与提示内容，使副屏向当前操作手提供可核验的战场事实、规则推断和本机模块异常状态。

本版本处理：

- 敌方复活方式推断及付费复活着重提示。
- 根据当前登录机器人身份、攻击增益和目标防御/易伤计算敌方斩杀线。
- 工程机器人使用撞击扣血阈值判断斩杀线。
- 本机模块首次离线、状态变化和恢复通知。
- 模块离线期间用持久模块状态面板临时替换主仪表盘侧面的事件时间轴。

本版本不处理：

- 指挥、语音、任务分配或“集火”等行动建议。
- 自定义图传、标准图传或解码链路调整。
- 操作面板及其他比赛操作功能修改。
- 新建第二套通知系统或重构全部战场数据为统一快照架构。

## 2. 规则与协议依据

规则来源为《RoboMaster 2026 机甲大师超级对抗赛比赛规则手册 V2.0.0（20260626）》：

- 第 5.2.2 节（PDF 第 76 页、正文第 75 页）：机器人战亡后复活读条每秒增加 1；位于补给区增益点或己方基地血量低于 2000 时，每秒增加 4。
- 表 5-2（PDF 第 68 页、正文第 67 页）：无增益时，42mm 弹丸对机器人装甲造成 200 点伤害，17mm 弹丸造成 20 点伤害，撞击造成 2 点伤害。
- 表 5-19（PDF 第 107 页、正文第 106 页）：攻击增益只提高弹丸伤害；防御/易伤改变弹丸攻击和撞击造成的伤害；增益计算后的最终伤害四舍五入。
- R56（PDF 第 142 页、正文第 141 页）：禁止主动冲撞对方机器人。因此工程斩杀线只是对规则定义的撞击扣血结果作状态提示，不构成冲撞操作建议。

协议来源为《RoboMaster 2026 机甲大师高校系列赛通信协议 V1.3.0（20260327）》：

- 第 2.2.12 节 `RobotModuleStatus`：模块状态 `0` 为离线，`1` 为在线，`2` 为因安装不规范被视为离线；各字段为可选字段。
- 第 2.2.14 节 `Buff`：消息提供机器人 ID、Buff 类型、增益值、最大剩余时间和剩余时间；类型 `1` 为攻击增益，类型 `2` 为防御/易伤。

所有复活方式均是根据血量变化、比赛时间和规则计算得出的推断，通知不得将推断描述为协议直接上报的事实。

## 3. 方案选择

采用在现有规则引擎上增量扩展的方案：保留 `NotificationRuleEngine`、各专项跟踪器、通知控制器、通知历史和用户设置，新增长效 Buff 状态与存在性明确的模块状态输入，并分别校正复活、斩杀线和模块判断。

未采用以下方案：

- 不把全部通知迁移到新的统一战场快照。该方案边界更整齐，但会扩大 v0.1.5 范围并迁移已经正常工作的逻辑。
- 不在每条 MQTT 消息到达时直接生成跨消息通知。复活、Buff 和血量需要组合多个时间点的数据，直接生成容易产生时序错误。

## 4. 数据流与状态边界

### 4.1 Buff 状态

通知消息入口增加对 `Buff` 的处理，并将 `topicBuff` 纳入通知功能必需订阅。Buff 缓存按 `robot_id + buff_type` 索引，保存增益值、协议剩余时间和消息时间。

- 同一键只接受时间不早于当前缓存的消息，避免乱序消息覆盖新状态。
- `buff_left_time` 为 0 时移除对应 Buff。
- 有效期截止时间为消息时间加 `buff_left_time`。
- 负的类型 2 增益值是有效易伤，不得作为非法数据丢弃。
- 类型、时间或数值无法安全解析时忽略该条消息，不中断 MQTT 数据流。
- Buff 缺失或过期时按无增益基准计算，但通知文案不得声称目标当前没有增益。

### 4.2 血量快照

`GlobalUnitStatus` 继续作为复活和斩杀线的触发输入。规则引擎在处理时同时读取：

- 当前登录机器人 ID，用于确定阵营和武器类型。
- 双方机器人血量。
- 比赛剩余时间和敌方基地血量。
- 当前仍有效的本机攻击 Buff 与敌方目标防御/易伤 Buff。

### 4.3 模块状态

模块状态输入必须保留 Protobuf 字段是否存在，不能再把未携带字段读取成默认值 `0` 后当作离线。状态跟踪和主仪表盘模块面板使用同一个存在性明确的模块状态模型，通知事件仍通过现有通知控制器发出。

### 4.4 状态清理

比赛切换、比赛结束（包括异常结束）、登录身份切换时清理复活计时、付费复活推断计数、斩杀线再触发状态和 Buff 缓存。MQTT 断开时清理 Buff 与模块状态基线；重连后的首个模块快照重新建立状态并可再次报告仍离线的模块。

## 5. 敌方复活方式推断

### 5.1 战亡记录

敌方机器人血量从正数变为 0 时记录：

- 战亡时间。
- 按现有可配置规则计算的本次所需复活进度 `P`。
- 普通免费复活时间 `P / normalProgressPerSecond`。
- 最快加速免费复活时间 `P / acceleratedProgressPerSecond`。
- 战亡期间敌方基地血量是否曾低于或等于 2000。
- 该机器人此前已被推断出的付费复活次数。

所需进度继续使用现有规则档案参数：基础进度、比赛已进行时间产生的进度、以及此前付费复活次数产生的进度惩罚。只在一次复活被推断为付费复活后增加该机器人的付费复活计数；数据不足或方式不确定时不增加。

### 5.2 分类边界

敌方机器人血量从 0 恢复为正数时，以实际战亡时长 `T` 和配置的时间容差分类：

- `T + 容差 < 最快加速时间`：推断为付费复活。
- 未达到普通读条时间，但不满足付费复活条件：推断为加速免费复活。
- 达到普通读条时间：推断为普通免费复活。
- 缺少战亡时间、比赛时间或其他必要输入：复活方式不确定。

加速免费复活的原因继续作为推断展示：战亡期间敌方基地曾低于或等于 2000 时说明“基地低血量加速”；否则说明“补给区加速”。

### 5.3 通知内容与级别

主标题统一为“敌方 X 号机器人复活”。详情统一以“敌方复活用时 X 秒，推断为……”开头。

- 推断为付费复活：使用 `CRITICAL` 级别，应用当前档案的高显著样式、声音、震动和展示时长，详情明确写“推断为付费复活”。
- 普通免费复活、加速免费复活和方式不确定：使用 `INFO` 级别。

己方复活通知保持现有事件类型，不套用敌方复活方式推断。

## 6. 斩杀线与 Buff 计算

### 6.1 当前机器人武器映射

只有“预计弹丸数”模式按当前登录机器人身份选择伤害来源：

- 英雄（1/101）：42mm 弹丸。
- 3 号步兵、4 号步兵、空中机器人、哨兵（3/103、4/104、6/106、7/107）：17mm 弹丸。
- 工程（2/102）：撞击扣血。
- 其他或无法识别的身份：可以提示目标进入斩杀线，但不显示预计弹量。

现有固定血量和血量百分比模式保持原有含义。

### 6.2 弹丸伤害

弹丸单发最终伤害按规则顺序计算，并只在最终结果处四舍五入：

`最终弹丸伤害 = round(原始弹丸伤害 × 本机攻击增益 × (1 - 目标防御增益))`

有效类型 1 Buff 的攻击倍率为 `buff_level / 100`，未取得有效攻击 Buff 时倍率为 `1.0`。类型 2 的防御值为 `buff_level / 100`；该值为负数时，`1 - 负防御值` 自然形成易伤倍率。预计弹丸数继续复用现有命中率和斩杀线阈值配置，但使用修正后的单发伤害。最终伤害小于或等于 0 时不生成弹丸数估计。

### 6.3 工程撞击阈值

工程不显示弹丸数。撞击最终伤害只计算敌方目标的防御/易伤：

`最终撞击伤害 = round(2 × (1 - 目标防御增益))`

敌方当前血量大于 0 且不高于最终撞击伤害时进入斩杀线。最终撞击伤害小于或等于 0 时不触发。

### 6.4 通知措辞

标题统一为“敌方 X 号机器人进入斩杀线”。详情按身份显示：

- 可识别的发射机器人：“预计还需 N 发弹丸”。
- 工程：“一次撞击扣血可清空当前血量”。
- 无法识别身份：只显示当前血量和进入阈值的事实，不显示预计弹量。

文案不得出现“集火目标”“建议攻击”等指挥性表达。现有冷却和重新进入阈值后的再触发机制继续保留。

## 7. 模块通知与持久状态面板

### 7.1 状态转换

- 状态 `0` 和 `2` 统一视为离线，标题使用“XX 模块离线”。
- `0` 与 `2` 互相切换不产生新通知。
- 状态 `1` 视为在线。
- 未携带的可选字段不改变现有状态，也不创建新状态。
- 未定义的状态值不覆盖最近一次有效状态。

每次连接后的第一个模块快照也参与通知判断：明确携带且为 `0/2` 的模块立即产生一次离线通知；首次为 `1` 不通知。在线变为离线时产生 `CRITICAL` 通知。离线恢复为在线时关闭对应持续离线提醒，并产生 `INFO` 级“XX 模块恢复在线”通知。

### 7.2 主仪表盘侧面板

主仪表盘侧面区域增加一个状态切换容器：

- 所有已监控模块在线时显示现有 `EventTimelinePanel`。
- 任一已监控模块离线时自动显示模块状态面板。
- 模块状态面板显示当前 MQTT 连接中至少被消息明确携带过一次的全部模块；后续消息暂时未携带某字段时保留其最近有效状态。离线模块置顶并高显著显示，在线模块以弱化样式显示。
- 全部已监控模块恢复为在线后自动切回事件时间轴。
- 时间轴隐藏期间，比赛事件继续由现有状态管理接收和保存；恢复显示后能看到隐藏期间产生的完整事件。

模块状态面板独立于“模块离线通知”设置。关闭模块离线通知只影响弹窗、声音、震动和通知历史，不影响持久模块状态面板。

## 8. 组件职责

- Buff 跟踪器：只负责接收、更新、过期和查询 Buff，不生成 UI。
- 复活跟踪逻辑：只负责记录战亡区间、计算边界和生成复活通知事件。
- `KillLineNotificationTracker`：根据当前登录身份和有效 Buff 计算斩杀线，保留现有冷却与再触发职责。
- 模块状态模型/控制器：保存存在性明确的模块状态，同时为通知转换和主仪表盘面板提供只读状态。
- `NotificationProtocolTracker`：根据模块状态转换生成离线/恢复事件，不保存 UI 状态。
- 主仪表盘侧面状态容器：只根据“是否存在离线模块”选择事件时间轴或模块状态面板，不参与协议解析和通知判定。

## 9. 错误处理

- 任何单条 Buff 或模块消息异常都只忽略该条无效字段，不得抛出异常中断 MQTT 消息流。
- Buff 过期、缺失或乱序时使用可解释的降级计算，不把未知状态表述为确定事实。
- MQTT 断线时不继续展示已经无法确认的模块离线面板；连接异常由现有连接质量通知负责。重连后以新快照重新判断。
- 复活分类缺少必要数据时显示“复活方式不确定”，不得默认推断为付费复活。

## 10. 测试与验收

### 10.1 规则单元测试

- 普通免费、加速免费、付费复活和方式不确定四类结果。
- 最快加速及普通读条边界的时间容差。
- 基地低血量与补给区加速原因说明。
- 只有推断为付费复活时使用 `CRITICAL`。
- 当前登录身份决定 42mm、17mm、撞击或无弹量说明，不能再由敌方目标身份决定弹丸类型。
- 攻击增益、防御增益、易伤、最终四舍五入、Buff 过期和乱序。
- 工程无增益、带防御、带易伤和最终伤害为 0 时的撞击斩杀线。

### 10.2 模块状态测试

- 首个快照已离线时提醒，首个快照全在线时不提醒。
- `0/2` 等价、二者切换不重复、恢复到 `1` 时关闭对应离线提醒。
- 未携带字段不被当作 `0`，未知值不覆盖有效状态。
- 断线清理基线，重连首帧可再次提醒。

### 10.3 Widget 与集成测试

- 任一模块离线时事件时间轴切换为模块状态面板。
- 离线模块置顶，在线模块仍可查看。
- 通知开关关闭时模块状态面板仍显示。
- 模块全部恢复后自动切回时间轴，隐藏期间新增的事件没有丢失。
- 现有通知历史、展示设置、固定血量模式和血量百分比模式回归通过。

### 10.4 完成标准

- `flutter analyze` 零问题。
- 相关最小测试和全量 `flutter test` 通过。
- `pubspec.yaml`、`feature_spec.md` 开发进度和附录 E 更新日志按正式开发版本要求同步为 v0.1.5。
- `feature_spec.md` 明确记录现有基础、实际改动、不处理范围和最终验证结果。
