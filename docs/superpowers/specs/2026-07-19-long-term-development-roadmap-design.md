# WOD Client 长期开发路线设计

日期：2026-07-19
当前正式版本：0.1.3
路线状态：候选版本已经逐项确认，尚未进入正式开发进度

## 1. 路线目的

本文档定义 WOD Client 在 0.1.3 之后的长期候选版本。路线围绕“当前机器人操作手的信息副屏”定位展开：向操作手提供公共战场信息、当前机器人状态、规则允许条件、快捷操作和异常通知，不承担队内指挥、语音沟通或任务分配。

本文档不预设 1.0.0 为项目终点。后续版本按独立功能持续追加。

## 2. 候选版本管理规则

1. 本文档中的版本号是路线排序，不代表版本已经启动。
2. 候选版本不得提前写入 `feature_spec.md` 的正式开发进度表或 Changelog，也不得提前修改 `pubspec.yaml`。
3. 某个版本正式启动时，才同步更新：
   - `pubspec.yaml` 的 `version`；
   - `feature_spec.md` 的版本块、Phase 和 Task；
   - 附录 E Changelog；
   - 发布时使用的 git tag。
4. 每个版本按一个主要功能划分，并明确现有基础、实际改动和不在本版本处理的内容。
5. 每个版本启动前仍须按 `feature_spec.md` 要求与用户确认具体 Phase 执行方式。
6. 规则、协议或外部项目接口变化时，可以重新调整尚未启动版本的范围和顺序。

## 4. v0.1.4 — 操作面板规则校正与协议状态接入

### 4.1 现有基础

- 已有按英雄、工程、步兵身份切换的操作面板。
- 已有常规买弹数量选择和 MQTT 指令发送。
- 已有工程开始兑换、确认装配和取消装配的定时发送逻辑。
- Protobuf 已定义 `can_remote_heal`、`can_remote_ammo` 和 `TechCoreMotionStateSync`。

### 4.2 实际改动

- 英雄常规兑换 42mm 使用 `CommonCommand.cmd_type = 2`，不再错误使用 17mm 的命令类型。
- 远程买弹参数使用远程兑换规则的最小单位 100 发，不套用常规兑换的 10 发下限。
- 删除英雄和步兵操作面板中的副屏复活确认按钮；复活操作继续由官方客户端完成。
- 远程回血和远程买弹分别由 `can_remote_heal`、`can_remote_ammo` 控制可用状态。
- 协议只表示不可用而没有提供原因时，界面只说明裁判系统当前不允许执行，不编造金币不足等具体原因。
- 按钮从不可用变为可用时播放一次短暂、低强度脉冲，不持续闪烁。
- 使用 `TechCoreMotionStateSync` 展示最高可选难度、科技核心运动状态、放入/平移/旋转步骤及剩余时间。
- 进入兑换流程后，按已确认策略持续发送确认装配；成功、取消或流程结束后停止。

### 4.3 验证重点

- 覆盖 17mm/42mm 命令类型与常规/远程兑换数量。
- 覆盖两个远程可用字段的独立转换和单次脉冲触发。
- 覆盖工程流程进入、成功、取消、结束、Widget 释放和身份切换时的定时器停止。
- 覆盖 `TechCoreMotionStateSync` 的非法范围与缺失状态降级。

### 4.4 不在本版本处理

- 不改主界面整体布局。
- 不调整通知语义。
- 不修改自定义图传。

## 5. v0.1.5 — 战场通知内容准确化

### 5.1 现有基础

- 已有全局通知覆盖层、严重级别、冷却、历史记录和规则档案。
- 已有敌方死亡/复活判定、斩杀线判定和模块离线/恢复事件。
- 已有通知设置和手动测试入口。

### 5.2 实际改动

- 敌方恢复存活时，主标题统一为“敌方 X 号机器人复活”。
- 次要描述显示复活用时，并把免费复活、付费复活和补给区加速复活明确标为推断。
- 证据不足时显示“复活方式不确定”，不能把推断写成协议直接事实。
- 斩杀线通知改为“敌方 X 号进入斩杀线，预计还需 N 发可击毁”，删除“集火目标”等指挥性表达。
- 击毁弹量按照当前登录操作手的武器类型选择 17mm 或 42mm，并计入己方攻击增益和敌方防御增益。
- 统一模块离线和恢复通知的标题、严重级别、去重键与历史记录行为。

### 5.3 验证重点

- 覆盖免费、付费、补给区加速和方式不确定的复活描述。
- 覆盖英雄、步兵和无射击能力身份的击毁弹量行为。
- 覆盖攻击/防御增益变化和模块在线状态转换。
- 保证首次模块状态快照不产生离线误报。

### 5.4 不在本版本处理

- 不重建现有通知运行时和设置系统。
- 不增加伤害统计面板。
- 不根据通知结果生成队伍指挥指令。

## 7. v0.1.7 — 赛时外部网络请求隔离

### 7.1 现有基础

- 已有 GitHub 比赛记录上传、下载、列表刷新和共享配置同步。
- 已有 GitHub Release 自动/手动更新检查。
- 已有实时 `GameStatus` 和比赛阶段状态。

### 7.2 实际改动

- 新增统一赛时网络请求守卫，由比赛状态决定 GitHub 和更新服务是否允许发送请求。
- 比赛进行中，服务层直接拒绝记录同步、配置同步和版本检查，不能只禁用界面按钮。
- 应用已经进入比赛后，即使 MQTT 暂时断开，也继续保持隔离。
- 只有收到明确比赛结束或返回赛前状态后，才恢复手动操作。
- 比赛期间相关入口禁用，并显示“比赛进行中，赛后可手动执行”。
- 应用启动但比赛状态尚未确定时，不立即执行自动更新检查。

### 7.3 验证重点

- 使用可注入网络客户端验证隔离期间向 GitHub API 发出的请求数量为零。
- 覆盖比赛开始、MQTT 断开/恢复、比赛结束和返回赛前状态。
- 覆盖自动检查和全部手动 GitHub 操作入口。

### 7.4 不在本版本处理

- 不重写 GitHub 同步和更新下载功能。
- 不禁止 MQTT 3333、UDP 3334 或本机解码桥通信。
- 不扩大到尚未确认的其他外部网络功能。

## 8. v0.1.8 — 比赛记录完整性与异常结束保存

### 8.1 现有基础

- 已有比赛消息记录、自动记录控制、JSON 导入导出和 Schema 版本。
- 已有记录列表、完整/未录到结算的基础判断。
- 已有本地回放和多文件合并。

### 8.2 实际改动

- 将记录结果扩展为“完整结束、异常结束、存在数据缺口”及其组合状态。
- 每场记录保存唯一标识、机器人身份、阵营、应用版本、协议版本、开始时间和最后一条有效消息时间。
- 比赛进行期间定期写入可恢复检查点；检查点写入失败时立即通知操作手。
- 正常收到比赛结束状态时，将检查点转换为正式完整记录。
- MQTT 长时间中断、比赛状态异常跳转或应用下次启动发现未结束检查点时，生成明确标记为异常结束的记录。
- 异常记录保存触发原因、最后有效数据时间和已知数据缺口，不能伪造正常结束时间。
- 应用被系统直接终止时，只保证恢复到最近一次成功检查点，不承诺恢复尚未写入检查点的内存数据。
- 记录恢复成功或失败均通过通知系统说明。
- 更新导入、记录列表、回放和合并逻辑，使其识别新的完整性字段，同时继续兼容旧 Schema。

### 8.3 BattleScope 兼容边界

`rm-battlescope` 当前主要读取官方 SQLite 数据集，其中：

- `matches` 为每局一行；
- `timeseries` 为每局、每秒、每实体一行；
- `events` 为每个离散事件一行。

WOD Client 当前以带时间戳的原始 Protobuf 消息列表保存数据，两者结构不同。因此本版本：

- 继续以原始消息记录作为权威数据，不直接替换为 BattleScope SQLite。
- 将记录器与导出格式分离，为以后增加 JSON、SQLite 或 BattleScope 适配器保留边界。
- 产出 WOD 字段到 `matches/timeseries/events` 的映射文档，区分直接转换、需要推断和协议没有的数据。
- 保留原始 Protobuf 数据，任何兼容导出都不能成为唯一数据副本。
- 在 `rm-battlescope` 的导入接口稳定前，不承诺具体兼容文件格式。

参考资料：

- <https://github.com/Zsdhak1/rm-battlescope/blob/main/README.md>
- <https://github.com/Zsdhak1/rm-battlescope/blob/main/docs/RMUC_2026_%E8%A7%84%E5%88%99%E4%B8%8E%E6%95%B0%E6%8D%AE%E7%B4%A2%E5%BC%95.md>

### 8.4 验证重点

- 覆盖正常结束、状态异常跳转、长时间断流、检查点写入失败和下次启动恢复。
- 验证异常记录不被误标为完整记录。
- 验证旧 Schema 导入以及新完整性字段的 JSON 往返。
- 验证检查点写入中断不会破坏上一份可恢复检查点。

### 8.5 不在本版本处理

- 不自动上传异常记录到 GitHub。
- 不直接把主记录格式替换为未经稳定约定的 BattleScope 格式。

## 9. v0.1.9 — 多客户端记录合并质量优化

### 9.1 现有基础

- 已有多文件导入和 `MatchMerger`。
- 已能检查阵营、比赛开始时间和最终比分是否一致。
- 已能按照 Topic、时间戳、机器人编号和载荷删除完全重复的消息。
- 合并记录已经保存来源文件和来源机器人编号。

### 9.2 实际改动

- 优先使用 v0.1.8 的稳定比赛标识判断记录是否属于同一局；旧记录继续使用时间、阵营和比分辅助判断。
- 根据各记录共有的比赛状态变化估算客户端墙钟时间偏差，并统一转换为相对比赛时间。
- 每条合并数据保留来源文件、来源机器人和原始时间戳。
- 完全相同的数据继续去重；内容冲突的数据全部保留，并标注冲突字段和来源。
- 生成按客户端、Topic 和比赛时间段统计的数据覆盖情况。
- 合并记录的完整性状态由所有来源共同计算，不能因单个来源完整而把整体标记为完整。
- 合并后生成新文件，不修改原始记录。

### 9.3 验证重点

- 覆盖时钟偏差、重复消息、状态冲突和部分来源中断。
- 覆盖新旧记录格式混合合并。
- 验证所有合并数据都能追溯到原始来源。

### 9.4 不在本版本处理

- 不重新实现现有导入导出和回放功能。
- 不加入赛后分析图表。
- 不自动从 GitHub 下载并合并全队记录。

## 10. v0.2.0 — 赛后分析看板

### 10.1 现有基础

- 已有记录列表、比赛回放、时间轴定位、倍速控制、事件列表和血量曲线。
- 已有多客户端记录合并，并由 v0.1.9 增加来源和完整性信息。
- 已有 `fl_chart` 图表依赖。

### 10.2 已确认范围

- 新增独立赛后分析入口，复用现有记录读取、状态重建和回放能力。
- 分析计算放在独立 Domain 层，界面只负责选择条件和展示结果。
- 每项结果区分协议直接记录、根据多项数据计算和规则推断。
- 推断结果显示依据、参数和不确定性，并能跳转到对应原始事件或回放时间段。
- 数据存在缺口时，相关分析降低可信度或停止计算，不能用缺失值补出确定结论。
- 单机记录和合并记录均可分析，但必须显示来源覆盖情况。
- 分析结果可以导出为独立报告，不能修改原始比赛记录。

### 10.3 启动约束

正式启动 v0.2.0 时，必须重新运行 `superpowers:brainstorming`，逐项确定：

- 分析事项；
- 计算公式；
- 图表形式；
- 筛选和交互方式；
- 报告结构；
- 每项推断的证据和置信度表达。

当前路线不提前固定经济曲线、事件分布或战斗关系等具体分析项目。

### 10.4 不在本版本处理

- 不重复开发播放、暂停、倍速和拖动回放。
- 不把赛后分析放入操作手赛中主界面。
- 不把推断结果写成裁判系统确认事实。

## 11. v0.2.1 — 全局比赛规则档案扩展

### 11.1 现有基础

- 已有通知规则档案、官方只读档案、自定义副本和 JSON 导入导出。
- 现有档案已经记录通信协议版本与比赛规则版本。
- 斩杀线、复活、部署跳转和连接质量参数已经可以配置。

### 11.2 实际改动

- 在现有通知档案基础上扩展为全局比赛规则档案，不建立平行档案系统。
- 将常规/远程兑换数量、价格和生效时间，复活规则，机器人伤害参数，攻击/防御增益计算等业务规则统一纳入档案。
- 支持同时保存多个官方规则版本；旧版本不因新规则发布而被覆盖。
- 比赛开始时保存本场使用的完整规则快照，并锁定本场规则；比赛进行中不能切换。
- 比赛记录写入规则档案 ID、规则手册版本、通信协议版本和参数副本。
- 导入档案时检查 Schema、规则版本和协议兼容性；无法确认兼容时禁止直接用于比赛。
- 迁移现有通知规则档案和用户自定义设置，不能要求用户重新配置。
- 规则档案与应用支持的 Protobuf 协议不一致时，明确提示需要更新应用。

### 11.3 模拟器训练规则

新增内置“模拟器训练”档案，并提供比赛完成判定方式：

- 正式比赛档案按官方比赛流程判断整场开始和结束。
- 模拟器训练档案把每一小局结束视为一场完整比赛结束。
- 单局结束时正常完成当前记录，不标记为异常结束。
- 保存该小局的开始时间、结束时间、比分、机器人身份和全部消息。
- 重置通知去重、复活计时和其他只在单场内有效的状态。
- 下一小局建立新的独立比赛记录。
- 回放、合并和赛后分析均把每一小局作为独立比赛处理。
- 模拟器训练档案明确标记为非官方规则。
- 完成方式由档案显式选择，不能根据 MQTT 地址或数据特征自动猜测。

### 11.4 验证重点

- 覆盖现有通知档案向全局档案的无损迁移。
- 覆盖官方档案只读、多版本并存和比赛中锁定。
- 覆盖规则快照随记录导出并在回放中恢复。
- 覆盖模拟器连续多小局分别生成完整记录并重置单场状态。

### 11.5 不在本版本处理

- 规则档案不能修改 MQTT Topic、Protobuf 字段编号或命令编号。
- 不允许通过 JSON 动态解释未知 Protobuf 消息。
- 不根据最新规则自动改写历史比赛记录。

## 12. v0.2.2 — 赛前联调与可用性检查

### 12.1 现有基础

- 已有 MQTT 登录和连接状态。
- 已有官方图传、自定义图传、解码器、记录目录和连接质量诊断。

### 12.2 实际改动

- 新增独立赛前检查页面，把分散诊断状态整理为一次可执行检查。
- 检查登录机器人身份、阵营、MQTT 连接和比赛必需 Topic 是否收到数据。
- 检查 UDP 3334 端口绑定、官方图传数据到达和解码器是否实际输出画面。
- 自定义图传只检查当前配置、数据到达和现有解码链状态，不修改其拼包或恢复逻辑。
- 检查记录目录是否可写、剩余空间是否达到阈值，以及检查点文件能否正常创建和删除。

## 13. 暂不编号的方向

### 13.1 自定义图传优化

现阶段不修改 `CustomByteBlock / 0x0310` 的拼包、丢弃或恢复策略。该链路是极低带宽、长 GOP、跨块 NAL 的连续码流，客户端随意删除字节会破坏 NAL 和帧间参考关系。

只有在取得真实链路样本并确认以下事项后，才重新启动 brainstorming：

- 当前固件实际包头和有效载荷格式；
- 序列号、长度前缀、填充和 `is_frame_start` 的可靠性；
- 丢包与发送端 backlog 裁剪的实际表现；
- SPS/PPS/IDR 周期及损坏后的恢复条件；
- 各平台解码后端在真实数据下的行为。

在上述证据具备前，不为该方向预占版本号。

### 13.2 RM BattleScope 格式适配

v0.1.8 只建立字段映射和导出边界。等 `rm-battlescope` 提供稳定导入约定后，再决定：

- 增加 BattleScope SQLite 导出适配器；或
- 提供双方约定的中间格式；或
- 在仍保留原始 Protobuf 的前提下调整主记录容器。

在接口稳定前，不为该方向预占版本号。

## 14. 跨版本工程约束

- 所有网络输入继续执行长度、范围和未知消息降级检查。
- 所有新增可导入/导出模型实现 `toJson/fromJson` 并携带 Schema 版本。
- Widget 不直接调用 MQTT、RawSocket 或文件系统服务。
- 所有 UI 字符串进入常量文件。
- 单函数不超过 50 行，异步操作包含错误处理和用户反馈。
- Windows、Linux 和 Android 采用同一纯 Dart 核心逻辑，平台差异只留在边界层。
- 每次文件写入后运行 `flutter analyze`；逻辑改动运行最小相关测试；发布前运行全量测试。
- 每个正式启动版本按 `vX.Y.Z Phase N, Task M` 登记，Phase 和 Task 从 1 重新编号。

## 15. 路线完成定义

本文档完成的含义是：v0.1.4 至 v0.2.2 的候选范围已经获得用户确认，并能在未来逐版转换为实施规格。它不表示这些版本已经开始开发，也不表示 v0.2.2 是项目终点。
