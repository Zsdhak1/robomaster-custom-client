# WOD Client 操作手信息副屏阶段性审查记录

日期：2026-07-18  
当前程序版本：0.1.3  
文档状态：记录截至本轮对话结束时，已经由用户逐项确认的结论

## 1. 文档目的

本文档用于保存对 WOD Client 现有功能、比赛规则对应关系和产品定位的阶段性审查结果，便于后续在压缩对话上下文后继续讨论。

本文档不是最终开发计划。后续仍按“一次只核实一个结论”的方式继续审查；尚未逐项确认的功能设想不得直接作为实现需求。

## 2. 产品定位

WOD Client 是当前机器人操作手使用的信息副屏，功能目的包括：

- 展示操作手在比赛中需要持续关注的机器人状态和公共战场信息。
- 根据规则状态判断某项快捷操作当前是否允许执行。
- 为官方客户端中操作路径较长、难以及时触发的操作提供快捷入口。
- 对需要操作手立即注意的异常、状态变化和比赛事件提供通知。

WOD Client 不承担队内指挥、语音沟通或任务分配。操作间已有队内语音，因此程序不应设计“第二指挥面板”，也不应使用“集火目标”等代替队伍指挥的表达。

## 3. 信息展示范围

比赛是多人协作场景，副屏不能只显示当前机器人自身的数据。信息分为两类：

### 3.1 所有操作手共享的信息

- 双方机器人血量和存活状态。
- 双方基地、前哨站等关键建筑状态。
- 双方经济和关键机制状态。
- 已确认的敌方复活等重大事件，以及后续逐项确认需要共享的关键机制信息。
- 比赛阶段、剩余时间和其他公共比赛状态。

### 3.2 随登录身份变化的信息

- 当前机器人能够执行的快捷操作。
- 只要求当前操作手采取行动的强提醒。
- 击毁弹量等依赖当前机器人武器类型和攻击能力的估算结果。

## 4. 当前主界面结论

当前主界面已经采用符合上述定位的分区结构：

- 顶部：登录身份、阵营和连接状态。
- 中部左侧：机器人状态和血量。
- 中部右侧：事件时间线。
- 底部：比赛状态、操作面板、记录状态和连接质量。
- 覆盖层：通知和英雄部署倒计时。

因此，现阶段不需要推翻主界面整体布局。后续审查重点是调整各区域显示的具体信息、优先级和占用空间。

## 5. 已确认的功能结论

### 5.1 英雄部署后进入自定义图传

该功能符合设计目标，应保留。

英雄进入部署模式后，标准 UDP 3334 图传会被切断，但机器人仍可通过未被切断的自定义数据部分发送图像。程序通过 MQTT `CustomByteBlock / 0x0310` 承载 H.264 字节块，为操作手提供替代图像。

因此，英雄部署状态触发后自动进入自定义图传页面，是为了在标准图传不可用时继续向操作手提供视觉信息，不是普通的页面自动跳转功能。

### 5.2 工程装配状态和自动确认

工程装配界面应由 `TechCoreMotionStateSync` 的真实状态驱动，展示装配进度、放入、平移、旋转和剩余时间。

操作手开始兑换并开启自动确认后，程序可以在兑换流程内持续发送协议定义的“确认装配”指令。其明确目的，是在机构第一次满足确认条件时立即完成确认，从而减少操作手持续观察和手动点击的负担。

持续发送应限定在兑换/装配流程内；装配成功、失败、取消或流程结束后停止。

### 5.3 英雄 42mm 弹丸兑换指令错误

当前英雄“兑换 42mm”功能发送 `CommonCommand.cmd_type = 1`，与协议定义不符：

- `cmd_type = 1`：17mm 弹丸。
- `cmd_type = 2`：42mm 弹丸。

这是明确的代码错误，后续实现阶段需要将英雄 42mm 兑换指令改为 `cmd_type = 2`。

### 5.4 常规兑换与远程兑换必须分别处理

两种兑换不是同一规则：

- 常规 17mm 弹丸兑换：最小兑换数量为 10 发。
- 远程 17mm 弹丸兑换：最小兑换数量为 100 发，价格为每 100 发 150 金币，兑换后 6 秒生效。

当前步兵“远程买弹”发送数量参数 10，不符合远程兑换规则。后续必须根据按钮实际功能使用远程兑换的数量限制，不能套用常规兑换的最小数量。

规则依据：比赛规则手册表 5-8、表 5-11，印刷页 78—79。

### 5.5 不在副屏重复提供复活确认操作

官方客户端已经提供复活操作。副屏应展示复活相关状态和事件，但复活确认继续在官方客户端完成。

因此，不新增副屏复活按钮，避免在两个客户端中重复提供同一项直接操作。

### 5.6 保留远程回血和远程买弹快捷操作

官方客户端的远程回血、17mm 弹丸兑换和 42mm 弹丸兑换位于按下 H 键打开的远程兑换面板内，还需要选择组合键并进行二次确认。

这些操作路径较长，不利于操作手在比赛中第一时间触发。因此，即使官方客户端已经具备相同能力，副屏的远程回血和远程买弹快捷按钮仍有明确价值，应保留。

按钮可用状态必须由协议字段控制：

- 远程回血：`can_remote_heal`。
- 远程买弹：`can_remote_ammo`。

不可用时按钮应禁用，并准确说明当前不能执行的原因。按钮由不可用变为可用时，在按钮周围播放一次短暂、低强度的脉冲光效，帮助操作手注意到状态变化；不持续闪烁。

官方客户端依据：选手端界面说明手册 PDF 第 15 页的 H 键远程兑换面板说明。

### 5.7 不增加伤害统计常驻展示

官方客户端会在机器人死亡后自动显示伤害统计。已确认的产品结论是：把累计伤害类型常驻在副屏上对操作手帮助有限，还会占用持续可见区域。

因此，副屏不增加伤害统计常驻面板。

### 5.8 模块离线提醒整合到通知系统

模块离线信息有直接行动价值，应保留，但不单独制作常驻模块面板：

- 模块由在线变为离线时，发送关键通知。
- 模块由离线恢复为在线时，发送恢复通知。
- 两类事件均写入通知历史。

这里的功能目的，是让操作手及时发现机器人模块故障及其恢复情况。

### 5.9 比赛期间停止 GitHub 同步和软件更新检查

比赛进行期间，程序不应执行 GitHub 同步或软件更新检查，以避免产生赛事服务器以外的网络通信。相关功能只允许在赛前或赛后由用户手动执行。

### 5.10 敌方复活通知

敌方机器人恢复存活时，通知主标题统一为：

> 敌方 X 号机器人复活

次要描述可以显示：

> 复活用时 X 秒，推断为免费复活/付费复活/补给区加速复活

复活方式属于根据用时和比赛状态计算的推断，不能作为协议直接上报的确定事实。无法可靠判断时，应显示“复活方式不确定”。

### 5.11 敌方斩杀线提示

敌方斩杀线提示有助于当前操作手判断是否能够完成击毁，应保留。但提示只陈述状态和估算结果，不替队伍作出指挥决策。

推荐表达：

> 敌方 X 号进入斩杀线  
> 预计还需 N 发可击毁

击毁弹量必须基于当前登录操作手所控制的机器人计算：

- 英雄按 42mm 弹丸计算。
- 步兵按 17mm 弹丸计算。
- 计算时考虑己方攻击增益和敌方防御增益。
- 不对所有登录身份使用同一个固定伤害值。

### 5.12 数据记录状态缩减为状态指示

数据记录正常时，不长期占用底部大面板，只显示小型状态图标。

当记录停止、存储失败或数据不完整时，通过通知准确说明发生的问题，使操作手能够判断本场数据是否仍被完整保存。

### 5.13 连接质量缩减为状态指示

连接正常时，不长期占用底部大面板，只显示小型连接状态指示。

当 MQTT 消息停滞、图传丢包率超过阈值或视频解码停止达到阈值时，通过通知说明具体故障类型。通知应描述可验证的问题，不能只显示笼统的“连接质量差”。

## 7. 后续讨论约束

- 每次只提出并核实一个结论。
- 每项结论必须明确说明功能目的、触发条件、显示内容或发送行为。
- 严格区分常规兑换、远程兑换及其他规则名称相近但条件不同的机制。
- 使用“写入通知历史”“保存为 JSON 文件”等可验证描述，不使用“落盘”等含义不明确的词语。
- 不因为官方客户端已有某项功能就直接判断副屏不需要；应结合官方入口的操作层级、比赛中的时间压力和副屏定位判断。
- 不把尚未获得用户确认的推测写成确定需求。

## 8. 当前阶段结论

现有程序的总体方向与“操作手信息副屏”定位基本一致，当前主要问题不是缺少另一个指挥系统，而是部分操作语义与规则不一致、底部记录/连接状态在正常情况下占用空间偏大，以及通知内容需要更准确地区分确定事实和推断结果。

后续应继续检查主界面各区域的具体信息价值，并在全部结论逐项确认后，再形成版本化开发计划。
